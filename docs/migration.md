## Key rehearsal findings (do not delete)

**division_id NULL after migrate:users**
During rehearsal: 373 users had division_id = NULL after migrate:users ran.
These were recovered by the restore pass built into migrate:users itself
(a second internal loop that re-matches users after the reconciliation
block). On a clean production run, 0 NULL division_ids are expected after
migration completes. Verify with:
  SELECT COUNT(*) FROM users
  WHERE division_id IS NULL AND is_super_admin = 0
  AND lifecycle_status != 'pending_engagement';
If non-zero: do NOT run fix:userdata until the source is identified.
The restore pass must complete before fix:userdata Fix 5 runs (Fix 5 deletes
users with NULL division_id AND organisation_id unconditionally).

**stats:backfill — lifecycle-NULL limitation**
Historical stat snapshots generated by stats:backfill may have NULL or
inaccurate lifecycle counts for past months. This is because the
scopeVolunteers() / scopeMembers() scopes evaluate lifecycle_status as it
stands at the time of backfill, not the historical point-in-time state.
This is a known limitation — document it but do not treat non-matching
historical counts as a migration error.

**lifecycle:reconcile**
lifecycle:reconcile [--apply] scans BOTH active and dormant users; promotes
AND demotes. This is the command to run post-migration and post-dates:shift.
With --apply it makes changes; without it is a dry-run. Also runs nightly via
the scheduler (lifecycle:reconcile --apply at 03:00). It does not scan
pending_engagement users.

(The older users:update-lifecycle-from-activity command — a one-way,
active-only, demote-only batch sweep — has been removed. It predated
lifecycle:reconcile and was never scheduled; lifecycle:reconcile fully
supersedes it, including the organisation-linked-user exclusion.)

## Production tail sequence (run in this exact order after migrate:old-db)

After migrate:old-db --table=all completes:

1. php artisan migrate:old-db --table=all   # (already run)
2. DEV ONLY: php artisan dates:shift N                # N = days since 2019-01-01
                                            # (2540 in the final rehearsal)
   

3. php artisan lifecycle:reconcile --dry-run
   # Record the output. Non-zero discrepancies are expected here because
   # the in-migration reconcile ran BEFORE dates:shift, against 2019 dates,
   # and promoted nobody. This dry-run is your baseline.

4. php artisan lifecycle:reconcile --apply
   # This is the operative reconcile — it runs against the shifted dates
   # and promotes dormant users to active where warranted. Must be run
   # MANUALLY here; it is not part of migrateAllTables() for this reason.

5. php artisan lifecycle:reconcile --dry-run
   # Should now report 0 discrepancies. Non-zero = investigate before
   # proceeding.

6. Verify lifecycle_status distribution:
   SELECT lifecycle_status, COUNT(*) as n
   FROM users GROUP BY lifecycle_status;
   Expected: meaningful active population. If pending_engagement is very
   large, decide whether a separate pass is needed (reconcile does not
   scan pending_engagement).
